# PRD：訪客/兒童地下城比照正式系統（T1-T2封頂＋裝備＋掉落物）

## 背景

訪客/兒童模式現在的地下城視覺跟正式學籍系統落差太大——`GuestDungeonSimple.jsx` 是完全獨立的簡化版（固定3層+固定王，`difficultyTier` 寫死1），跟正式系統的迷霧格子探索/前後衛編隊/裝備加成完全沒有關聯，UI 風格也是另外刻的 inline style，跟品牌質感對不上。

目標：訪客/兒童的地下城改成**直接使用正式系統的元件與邏輯**（不是再做一套模仿），但用參數把難度封頂在 T1-T2、隱藏還沒開放的功能區塊，並開放裝備系統與真實掉落物，讓訪客/兒童帳號的遊戲體驗接近正式學籍系統。

**這是在使用者睡覺期間自主規劃執行的任務，以下標記「〔自主決定〕」的項目是我依研究結果自己拍板、沒有跟使用者逐項確認，使用者醒來後應該review這些決定是否符合預期。**

## 範圍

### 核心整合
1. **`DungeonLobby.jsx` 開放給訪客/兒童使用**——新增可選的 `guestProfile`／`isGuest`／`tierCap` 參數（沒傳就照舊呼叫 `useAuth()`，正式學生行為完全不變）。訪客/兒童模式呼叫時傳入自己的 `profile`（guest-kid-mode-overhaul 已經是真實持久化的 `members` 文件）與 `tierCap=2`。
2. **難度封頂 T1-T2**——所有原本讀 `excavation.difficulty`（T1-T6）來決定怪物/寶箱/王的地方，訪客/兒童模式一律夾在 1-2 之間，不能生出T3以上內容。
3. **隱藏「挖掘探索」分頁**〔自主決定〕——正式系統靠每日射箭量賺取T1-T6解鎖機率，訪客/兒童是後台直接鎖定不是「賺」出來的，這個分頁對他們沒有意義，整個不顯示（比照現有「隱藏不是灰階」的既有慣例，`accessControl.js`/`MonsterBattle.jsx` 已經是這個模式）。
4. **只做單人遠征，不做組隊**〔自主決定〕——組隊涉及房號建立/加入/斷線重連，複雜度顯著更高，這次先不做；訪客/兒童的地下城入口只給「進入地下城→單人遠征」這條路徑。
5. **開放裝備系統**——`EquipmentPage.jsx` 同樣加可選 `guestProfile` 參數，讓訪客/兒童能看到並升級自己的 `rpgEquip`。研究確認這個頁面目前完全沒有 `isGuest` 判斷、也沒有 Firestore 規則白名單限制訪客寫入（guest/kid 分支不受 `hasOnly` 限制），風險集中在前端資料流，不在規則層。
6. **實裝真實掉落物**——地下城戰鬥結束依 `lootTable.js`/`monsterMaterials.js` 現有邏輯發放材料/金幣，寫回訪客/兒童自己的 `members` 文件（比照正式系統，不要另外發明一套訪客專屬材料表）。**不動 `MonsterBattle.jsx` 既有的 `LOOT_TABLE_GUEST`（教練現場發實體勳章那套首勝流程）**〔自主決定〕——那是還在運作中的實際營運流程，這次範圍只有地下城，不去動打怪首勝的既有邏輯。

### UI/質感
7. 訪客/兒童地下城 UI 直接使用正式系統的元件（`DungeonExpedition.jsx`/`TeamExpeditionBattle.jsx` 內的單人遠征部分/`DungeonSelectionPanel.jsx`），不要再刻一套 inline style——這樣「質感落差」問題自然解決，因為用的就是同一套視覺。
8. 圖鑑（`DungeonDex.jsx`）〔自主決定：保留顯示，不特別過濾〕——訪客/兒童能看，內容就是他們實際拿到的材料，不去額外處理「T3-T6內容要不要在圖鑑裡隱藏」這種深層過濾，範圍太大，這次先跳過，圖鑑本來就是唯讀展示，風險低。

## 技術決策（詳見 design.md）
- 整合方式：明確 prop 傳遞（`guestProfile`/`isGuest`/`tierCap`），不用 Context 包裝偽裝成正式登入身份——維持這個專案既有的「`isGuest`/`kidMode` 明確傳遞」慣例，不引入新模式。
- 隱藏而非灰階：延續 `accessControl.js`/`MonsterBattle.jsx` 現有的「條件式不渲染」寫法，不要新發明一套「disabled狀態」UI。
- 舊的 `GuestDungeonSimple.jsx`：完全整合完成、驗證過後刪除；過渡期間可以先並存，實作時再視情況決定要不要中途刪除。

## 驗收
1. 訪客/兒童帳號進地下城分頁：看不到「挖掘探索」分頁，只有「進入地下城」（單人）＋「圖鑑」。
2. 訪客/兒童能實際跑完一場單人地下城遠征（迷霧格子探索、前後衛編隊、王關），視覺跟正式學生看到的一致。
3. 遠征只會抽到 T1 或 T2 難度的怪物/王，不會出現 T3 以上內容（程式碼層面確認 `tierCap` 有確實夾住所有相關的隨機抽取函式，不是只擋UI）。
4. 訪客/兒童能進裝備頁看到並升級自己的裝備。
5. 打完地下城後，訪客/兒童的 `members` 文件裡材料/金幣有正確增加（讀真實 `lootTable.js` 邏輯，不是假資料）。
6. `MonsterBattle.jsx` 既有的訪客首勝實體勳章流程完全沒被動到（regression 檢查）。
7. 教練切換射手模式後，動到的正式系統元件（`DungeonLobby`/`EquipmentPage`/`DungeonExpedition`）不能白屏——這幾個元件現在被訪客模式共用了，任何改動都要對正式學生端做一次完整檢查，這是本次風險最高的部分。
8. `CI=true npx react-scripts build` 通過。

## 非目標（本次不做）
- 訪客/兒童組隊地下城。
- 動 `MonsterBattle.jsx` 既有訪客首勝實體勳章流程。
- 圖鑑內容按 tier 過濾隱藏未達成項目。
- 挖掘探索系統開放給訪客/兒童（連分頁都不會出現）。
