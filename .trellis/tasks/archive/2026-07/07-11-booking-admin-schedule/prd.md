# 線上約課後台排程與預約流程改善

## Goal

讓教練進入線上約課後台時，能以日檢視快速掌握每個實際靶位的占用狀態、一次關閉連續區域時段，並從已點選的時段直接建立不受學生端時間限制的代客預約。

## Requirements

1. 行事曆進入時預設使用日檢視，仍保留切換週檢視的能力。
2. 行事曆的視覺占位必須反映實際 `participantCount`：同一筆預約 2 人就顯示 2 個占位格，3 人就顯示 3 個占位格，不得只顯示一筆卡片而必須點詳情才知道人數。
3. 2 或 3 小時預約必須在時間軸上呈現連續占用，不應被誤認成各小時互不相關的獨立預約。
4. 教練可一次選擇日期與開始／結束時間，批次關閉或重新開放整段連續時段，不必逐小時操作。
5. 從某個時段點擊「在此時段新增預約」後，建立流程固定沿用該時段作為開始時間；搜尋並選定顧客後，只顯示方案、時數、人數與必要確認資訊，不再要求重新選日期／時間。
6. 方案時數改變時，結束時間由已點選的開始時間自動延伸；若跨出的任一格已滿或封鎖，必須顯示明確錯誤且整筆不建立。
7. 教練代建預約可建立過去日期／時間，也可建立距現在少於 30 分鐘的預約；學生自助與公開自助入口仍維持既有 30 分鐘前置規則。
8. 所有多人與跨時段容量仍須透過既有 Firestore transaction 原子檢查與寫入，不得用多筆循序預約模擬。
9. 既有單一時段封鎖／解除能力可保留，但區域時段操作需成為明確可用的主要批次工具。

## Acceptance Criteria

- [ ] 初次進入行事曆顯示今天的日檢視。
- [ ] 一筆 3 人預約在每個占用時段清楚呈現 3 個實際占位格，總容量仍為 8。
- [ ] 一筆 2／3 小時預約在日檢視呈現為連續的時間占用，姓名、方案、人數與時數不需進詳情即可辨識。
- [ ] 教練可一次關閉例如 13:00 至 17:00 的所有小時格，且可一次重新開放同一區間。
- [ ] 區域操作部分失敗時不得留下難以判斷的半完成狀態；UI 必須回報結果。
- [ ] 從 14:00 格新增預約、搜尋顧客後，不再出現日期／時段選擇器。
- [ ] 在上述流程選 3 小時後，自動建立 14:00 至 17:00 的預約並原子占用 14／15／16 三格。
- [ ] 教練可替顧客建立昨天或更早的預約紀錄。
- [ ] 學生端與公開預約端仍拒絕過去或少於 30 分鐘前置的時段。
- [ ] 正式版建置通過，並完成日檢視手機與桌面版的排版檢查。

## Constraints

- `bookingSlotCounts` 仍是容量與封鎖狀態的唯一來源。
- `bookings.slotKeys[]` 仍代表預約實際占用的連續小時格。
- 管理員豁免必須是明確資料層參數或專用管理員函式，不可全面移除 `checkLeadTime`。
- 不改變學生可見的其他顧客隱私邊界；姓名與方案只出現在教練後台。

## Confirmed UX Decision

- 日檢視採 8 靶位時間軸；多人預約橫向占用相對應靶位數，2／3 小時預約由開始時間向下跨越 2／3 個小時列，形成單一連續長區塊。
- 週檢視保留精簡摘要，不展開成七天乘八靶位的完整排程板。
