# PRD：預約系統 UI 優化——人數多靶位原子鎖定＋確認頁＋卡片版面

## 背景

使用者實測前台約課流程後回饋一批 UI/UX 問題，逐項處理。

## 已確認的設計決定
- **人數選擇＝真的多佔位**（使用者拍板）：選 N 人＝這一筆預約在同一個時段原子性地鎖定 N 個靶位，不是備註用的假欄位。時段容量不夠 N 人時，那個人數選項要直接變成不可選（不是選了才報錯）。
- 人數是**整筆預約共用同一個 `isNewStudent`／`planType`**（不細分「這N人裡面誰新誰舊」），這是本次刻意簡化，如果要拆分要另外討論。

## 範圍

### 1. `bookingDb.js` 容量交易推廣到「每格可佔用 N 個名額」
現有的多時段（1/2/3小時）原子鎖定已經是「對 slotKeys 陣列裡每一格做讀取→檢查→寫入」，這次再疊一層：每一格檢查的不是「count 是否 <8」，而是「count + participantCount 是否 <=8」；寫入時每一格 `count`（與對應的 newCount/returningCount）都要 `+participantCount`，不是固定 +1。取消/改期同理，要用 `booking.participantCount`（不是永遠當作1）去釋放/搬移對應數量的名額。

### 2. 時段選擇要感知目前選的人數
`bookingSchedule.js::slotState()` 要多吃一個 `participantCount` 參數，判斷「這個時段（含它涵蓋的每一格，3小時方案要連續3格都要有空間）扣掉已佔用的，是否還能塞得下這麼多人」，塞不下要直接 disabled 並顯示「人數超過剩餘名額」，不是選了才在送出時失敗。

### 3. 選時段 → 先「確認預約」畫面 → 才進註冊/登入
現況：點時段格 → 直接進註冊/登入表單。改成：點時段格 → 先跳出一個「確認這個時段」的畫面，完整顯示：日期時間、方案類別、時數、人數、**總金額（單價 × 人數，自動加總）**，使用者按「確認」才進到下一步（新客走註冊、舊客走登入）。這個確認畫面同時也是使用者反映「已選擇只顯示日期時間，看不到人數/方案/金額」的修正點——不管是這個確認畫面，還是任何後續顯示「已選擇」的地方，都要完整顯示這些資訊，不能只有日期時間。

### 4. 方案選擇改成並排小卡樣式
`PlanDurationPicker.jsx` 現在是 `grid-cols-1`（直向堆疊的長條列表），改成類似 `BillingSystem.jsx`/`CheckoutModal` 那種並排小卡片（`grid-cols-2` 或 `grid-cols-3`，視方案數量與螢幕寬度決定），每張卡片顯示方案名稱+時數+金額，選中要有明顯的視覺區別。

### 5. 人數選擇 UI
新增一個人數選擇元件（1~8人的步進器或按鈕群組，具體上限依 `LANE_CAPACITY` 決定，不要寫死8這個數字，直接引用常數），跟 `PlanDurationPicker` 一樣要感知目前選的時段還剩多少名額（互相連動：選人數會影響哪些時段能選，選時段後人數選項如果超過剩餘名額也要跟著收斂）。

## 待討論（本次不直接實作，先收集想法）
使用者提到想「增加一些可填寫欄位」跟「預約注意事項」，這兩塊還沒定案，這次先不做，留到確認具體要什麼欄位/文字內容後再排。

## 驗收
1. 選3人的3小時方案，若其中任一格扣掉3人後會超過8人上限，這個人數選項要直接不可選，不能選了才在送出時報錯。
2. 點時段格後看到完整的「確認預約」畫面（日期/時段/方案/時數/人數/總金額），按確認才進註冊/登入。
3. 方案選擇是並排小卡片，不是直向長條列表。
4. 取消一筆3人的預約，對應時段格的容量正確一次釋放3個名額（不是只釋放1個）。
5. `CI=true npx react-scripts build` 通過。

## 非目標
- 不做「同一筆預約裡拆分新生/舊生人數」的細節設計。
- 不做本次「待討論」的新增欄位／注意事項內容（等討論完再排）。
