# 技術設計：地下城緊急復原

## 範圍與根因

`DungeonBattleRoom` 新增的共用 `BattleScreen` 分支在元件執行期間先讀取 `myWbFrame`，但其 `useMemo` 宣告位在該分支之後。`const` 尚處於暫時性死區，因此進入 `active` 或 `resolving` 狀態即拋出 ReferenceError。

地下城房間的建立資料由 `DungeonLobby` 的 `buildExpeditionMemberData` 產生，已將已裝備卡片的 HP、ATK、DEF 與貓咪資料寫進隊員資料。新版 `BattleScreen` 適配層卻將目前玩家的 `cat` 固定傳為 `null`，且沒有完整傳遞貓咪 XP、羈絆與類型；因此共用戰鬥畫面無法重建貓咪戰鬥／羈絆效果。

## 修復方向

1. 將世界王卡框的衍生值移到所有條件式 return 之前，維持 Hook 順序並消除 TDZ。
2. 保留房間中已持久化的卡片加成數值，不在戰鬥畫面再次相加，以避免重複計算。
3. 讓遠征成員資料從大廳、單人戰鬥房、組隊協調房到每層組隊戰鬥房，都持久化足以重建貓咪戰鬥資料的欄位（id、名稱、類型、XP、羈絆）；新版 `BattleScreen` 對目前玩家傳入該資料。資料缺失時使用安全預設值，兼容舊房間。
4. 單人與組隊都走同一個 `DungeonBattleRoom` 適配層；針對目前玩家驗證共用 `BattleScreen` 的貓咪效果，其他隊員只作資料展示。

## UI 映射修正

- 單人地下城使用冒險單人打怪的 `BattleScreen` 視覺契約；組隊地下城使用組隊打怪的 `BattleScreen` 視覺契約。
- 地下城仍保有伺服器權威的回合提交、結算與獎勵，畫面不得自行重算地下城傷害。
- 共用畫面以 renderer props 接收地下城既有的怪物與射手資產來源，不能退回不存在的 avatar 或 emoji fallback。

## 不變條件

- 不改變 Firestore 已建立房間的 HP／ATK／DEF，也不遷移玩家資料。
- 卡片數值只在建立／加入遠征時帶入一次。
- 僅限地下城復原相關檔案，避免碰觸其他未提交變更。

## 風險與回復

- 舊房間缺少新貓咪欄位時，將退回既有無貓咪資料的安全顯示，而不阻斷進場。
- 若驗證顯示 `BattleScreen` 的貓咪介面與地下城既有規則不相容，保留資料持久化修復，並將效果層改為最小、相容的適配方式。
