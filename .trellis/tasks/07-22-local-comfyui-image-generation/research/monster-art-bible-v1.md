# 怪物美術規格 v1

## 1. 核心原則

- 每個 monster ID 只有一張透明高解析 canonical master art。
- 戰鬥立繪直接使用完整母版；卡片使用同圖 3/4 身裁切並合成角色專屬場景；圖鑑使用半身裁切；地城使用同角色的縮放／等角合成。
- 禁止為卡片、戰鬥、圖鑑與地城分別自由生成不同角色。
- 怪物、卡片與場景必須共享角色輪廓、臉部／頭部特徵、服裝、配色、代表性道具與材質語言。
- 主視覺以好看、角色關聯與辨識度為優先；不強制持弓、持箭或武器。

## 2. 人形與純怪物

- 男／女人形：6～7 頭身，成熟日系 RPG 2D 插畫；漂亮、帥氣、華麗，非照片寫實、非 Q 版、非 3D 公仔。
- 純怪物：比例自由，可四足、飛行、無肢體或異形；輪廓易辨識，保持相同線條、上色與光影媒材。
- 禁止 gore、blood、body horror、rotting flesh、exposed organs、photoreal horror、grotesque deformity。

## 3. 角色槽配比

### 普通組（所有 Tier 固定，不輪動）

| Slot | 型態 |
|---|---|
| `normalA` | 男 |
| `normalExisting` | 女 |
| `normalB` | 純怪物 |

既有普通怪外觀全部重新設計；舊名稱、技能、材料與主題只作概念參考。

### 首領組（T1～T6 輪動）

| Tier | `miniA` | `miniB` | `boss` |
|---|---|---|---|
| T1 | 男 | 女 | 純怪物 |
| T2 | 女 | 純怪物 | 男 |
| T3 | 純怪物 | 男 | 女 |
| T4 | 男 | 女 | 純怪物 |
| T5 | 女 | 純怪物 | 男 |
| T6 | 純怪物 | 男 | 女 |

每個 Tier 的普通組與首領組都各自保持 1 男、1 女、1 純怪物；每個 Tier 合計 2:2:2。

## 4. Tier 華麗度

| Tier | 定位 | 規則 |
|---|---|---|
| T1 | 初階 | 乾淨輪廓、1～2 主色、簡潔服裝與基礎代表物；仍須漂亮／帥氣 |
| T2 | 熟練 | 第二材質、局部紋樣、精緻配件與更完整場景關聯 |
| T3 | 菁英 | 多層服裝、種族徽記、明確職業輪廓、少量受控能量效果 |
| T4 | 高階領袖 | 華麗披風／甲冑、稀有材質、複合道具與更強輪廓張力 |
| T5 | 傳說 | 獨特神器、精細飾品、種族象徵環繞、受控大型光效 |
| T6 | 神話 | 標誌性完整造型、最高級材質與神器、強烈但不遮臉的神話光環 |

升階依靠服裝、材質、配件、道具與輪廓；不得只堆粒子特效。

## 5. 種族方向

- 鬼怪：全球幽靈／亡靈文化，可取材世界各地服裝與傳說，避免無意義文化拼貼；不綁台灣民俗。
- 山林：動物系為主。
- 毒蟲：華麗昆蟲幻想；人形融入翅膀、甲殼、觸角與毒素色彩；怪物採幻想化節肢輪廓，禁止寄生與寫實蟲恐。
- 職場：保留現代職場題材。
- 考試：保留現代考試／學術壓力題材。
- 神殿：失落文明守護者；祭司、聖騎士、占星師、神獸、魔像與機關生命。可取材古文明，不直接複製現實宗教神祇與儀式。
- 寶藏：冒險、鍊金與活化財寶；禁止賭場／拉霸與現代炫富語言。

## 6. 卡片規格

- Asset 比例固定 3:4。
- WebP 只包含 canonical 角色裁切與角色專屬場景；卡框、名稱、Tier、技能、星級與數值由 React UI 疊加。
- 禁止圖片內文字、數值、卡框與 UI。
- 同一 `family × tier` 共用色彩、材質、光線、紋樣與華麗度，不共用完全相同背景。
- 場景需呼應個性、技能或故事，但不得搶走角色焦點。
- 卡片角色通常採 3/4 身裁切；實際姿勢、道具與構圖依角色設定，不要求武器完整入鏡。

## 7. 內容重寫邊界

可以重寫：怪物顯示名稱／稱號、技能顯示名稱／摘要／反制提示、掉落物顯示名稱／說明、卡片／圖鑑／手冊敘事。

必須保持：monster/material/skill ID、資料 schema、HP/ATK/DEF、掉落數量、轉換規則、技能公式、倍率、機率、回合數、觸發條件與實際效果。

## 9. 生成品質閘門

- 先完成人工批准的角色 spec，再生成 canonical master；不得由模型自行決定角色身份。
- 生成 metadata 必須保存 checkpoint、LoRA、workflow version、prompt、negative prompt、seed、sampler、scheduler、steps、CFG、尺寸與輸出 hash。
- 同角色各衍生圖必須通過 identity comparison。
- 未批准輸出只留在 Git/Vercel 排除的 staging，不得覆寫 `public/`。
- 本機模型／workflow 無法達成主視覺時，保存失敗證據後由使用者明確批准才可轉交 GPT；禁止自動 fallback。
