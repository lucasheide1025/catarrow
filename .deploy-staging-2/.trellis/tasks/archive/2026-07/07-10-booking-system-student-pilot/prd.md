# PRD：線上約課預約系統（學生試用版）

## 背景

現況：官網「立即預約」CTA 導去第三方 SimplyBook，跟這個 App 的學籍系統（`members` collection）完全沒有資料串接。本次要做一套自製預約系統，跟 SimplyBook **並存**——不是取代，是先開放給既有學生試用，驗證穩定後才考慮把官網 CTA 換過來（換連結那一刻的事，不在本次任務範圍）。

並存分工（本次任務範圍）：
- **官網訪客**：官網 → SimplyBook（完全不動，這次不碰 `website/`，官網上不會出現任何連到新系統的連結）
- **在校學生（已有帳號）**：`MemberApp.jsx` 新增「線上約課」分頁 → 新系統
- **新生（還沒有帳號）**：**新增一個公開自助註冊+預約入口**，重用既有訪客模式 `resolveGuestSession()` 機制（匿名登入＋email/電話找回或建立記錄），註冊完直接進預約流程。**這個入口這次刻意不對外公開連結**（不放官網、不進主導覽選單），只給教練一個未公開的直接網址，測試階段自己掌握要給誰用，等於是使用者要求的「隱藏按鈕」——整個入口本身就是那顆隱藏按鈕
- **電話進線**：教練後台代建，跟上面兩種入口共用同一套底層 `createBooking()` 邏輯

⚠️ **風險提醒（不要跳過）**：公開自助註冊入口＝不需登入就能寫入 Firestore 的端點，是今天稍早處理 Firestore 額度事故的同一種風險類型（差別是那次是auth身份混淆的bug，這次是單純流量/濫用風險）。這次的防護只靠「網址不公開」這一層，沒有做 App Check 或其他反濫用機制——如果之後這個網址被分享出去或這套系統要真的公開上線，必須先補上更完整的防護（見「非目標」）。

## 上線策略（明確要求，不要自作主張）

- **這次做完先不要 push 到 `main`**。這個 App push main 會自動觸發 Vercel 部署到正式站，跟 `website/` 的手動部署流程不一樣，風險層級更高。
- 全部功能（含學生分頁、新生隱藏入口、教練後台）都做完、`CI=true npx react-scripts build` 通過、且使用者自己測試過沒問題之後，才問要不要 push。
- 即使 push 上線，`MemberApp.jsx` 的「線上約課」分頁預設也應該只對有 `bookingBetaAccess:true` 旗標的會員顯示（見 design.md），不是上線就對所有學生開放——這是額外一層漸進開放的保護，教練可以自己控制先開給誰。

## 已確認的規則（討論過程逐項拍板，不要重新討論）

| 項目 | 決定 |
|---|---|
| 場地容量 | 全場固定 **8 個靶位**。線上自助預約上限＝8 個並行名額（一人一靶一時段） |
| 團康（8-16人） | **不走線上自助**，維持電話／LINE 洽詢，教練用後台行事曆手動建立，系統不用理解「兩人共用一靶輪流」這種人為調度邏輯 |
| 多教練 | 支援，但**教練不是客戶可選項**——客人只選時段，教練分配是後台排班/會計用的內部欄位，不影響容量計算（容量是全場 8 靶位，不是按教練分） |
| 時段封鎖 | **全場封鎖**（封鎖某時段＝整個8靶位那個時段都不開放），不做分教練封鎖 |
| 變更時段（改期）／取消 | **學生自己可以在前台隨時操作，不設緩衝限制**——因為沒有收訂金，臨時取消/改期沒有金錢損失風險，不需要用時間限制去保護什麼 |
| 預約/改期的最短前置時間 | **新增規則**：不能預約或改期到「距離現在不到 30 分鐘」就要開始的時段（例如現在 10:00，最早只能約 10:30 開始的時段）。UI 上這類時段要直接顯示不可選，並在使用者嘗試選擇太近的時段時給明確提示說明原因 |
| 金流 | 不開放線上收款/訂金，這次不碰 |
| 營業時段 | 沿用現有時間：週一公休，週二 13:00-22:00，週三～週日 10:00-22:00 |
| 顧客欄位 | 姓名／Email／電話，**Email 與電話必填**（是轉正式學籍的依據，見下方資料模型） |
| 電話進線預約 | 教練在後台手動建立，可指定人數/方案，跟線上自助走同一套底層寫入邏輯 |

## 核心功能範圍

1. **新增預約**——學生前台自助 + 新生隱藏入口自助註冊+預約 + 教練後台手動建立（含電話進線），三種入口共用同一套底層邏輯。
2. **查看預約**——教練後台行事曆，週/日視圖切換（比照現有 SimplyBook 後台的視覺習慣：色塊呈現、可切換視圖粒度）。
3. **取消預約**——雙邊都能取消（學生取消自己的、教練可取消任何一筆），取消要釋放掉該時段占用的名額。
4. **變更時段**——學生前台自助（無緩衝限制，但新時段要滿足30分鐘最短前置時間），教練後台也能改。
5. **暫停特定時段預約**——教練後台功能，全場封鎖某時段。
6. **顧客「更多紀錄」摘要**——只顯示：最早預約時間、累計預約次數、最近一次預約時間。**這三個欄位必須是預先算好存在會員文件上的欄位，不能是每次開列表都重新查詢預約紀錄算出來的**（避免顧客一多，列表頁變成讀取熱點——今天處理 Firestore 額度問題時的同一個教訓）。「累計預約次數」語意＝**目前有效預約數**，取消要扣回去（不是「歷史上總共建立過幾次」）。
7. **後台收費方式與分類報表**——依方案類別（單人一般／兒童學生敬老／自備器材）× 收款方式（現金／轉帳）統計，供教練記帳用，是新報表，不沿用既有的 `AdminFinance.jsx`（那個是月費卡用的）。

## 資料模型與技術決策（詳見 design.md）
- 新 collection：`bookings`。
- 容量計算用 `runTransaction` 對「日期+時段」做原子計數，不能用「先查後寫」（這個 App 的鐵律之一，多人搶同一時段時必崩）。
- 客戶資料沿用既有 `members` collection 與 `accountType`／`contactHash` 機制——電話進線的顧客可以先建一筆低權限的會員記錄，之後要轉正式學籍時走既有的 `convertGuestToOfficial` 同一套邏輯，不要另建一套顧客資料表。

## 驗收
1. 學生能在 `MemberApp.jsx` 新分頁完整跑一次：選時段 → 送出預約 → 看到自己的預約 → 改期一次 → 取消一次。
2. **新生能透過隱藏網址完整跑一次**：填姓名/email/電話 → 選時段 → 送出預約成功，且這個網址不會出現在官網或主 App 導覽的任何地方（grep 確認沒有任何可見連結指向它）。
3. 教練能在 `AdminApp.jsx` 新分頁：看週/日行事曆 → 手動建立一筆預約（含電話進線情境）→ 封鎖一個時段確認該時段容量歸零 → 看收費分類報表。
4. 兩人（開兩個分頁模擬）同時搶同一個時段的最後一個名額，只有一人搶得到，不能兩人都搶到（容量 transaction 正確性核心驗收）。
5. 距離現在不到30分鐘的時段，三個入口（學生/新生/教練代建）都要正確擋下並給出提示，不能只有前台UI擋、後端沒擋（後端也要驗證，UI擋只是體驗優化，不能當唯一防線）。
6. 顧客列表頁的「更多紀錄」三欄位效能驗收：肉眼/程式碼確認是讀會員文件本身的欄位，沒有對 `bookings` collection 額外查詢；取消一筆預約後確認「累計預約次數」正確扣回去。
7. `CI=true npx react-scripts build` 通過。
8. Firestore 規則變更（新 collection 的讀寫規則）明確提醒使用者手動貼 Firebase Console。
9. **完工後，主動跟使用者複述一次「這個隱藏網址是什麼、目前只有誰知道」**，確保部署完成當下沒有人不小心把它公開分享出去。

## 非目標（本次不做，含明確排除的理由）
- 官網 CTA 換成新系統——這次只是試用版，SimplyBook 保持現狀，官網不連結新系統任何入口。
- **Google／Facebook／LINE 第三方登入**——這次試用階段的三種入口都不需要（學生已有帳號、新生走隱藏連結的contactHash機制、教練是既有admin登入），留到未來這套系統真的要公開給一般大眾取代SimplyBook那天，跟「新生隱藏入口要不要加強防濫用機制（App Check等）」一起評估、一起做，不要現在半套做。
- 線上金流/訂金。
- 團康（8人以上）線上自助預約——維持電話/LINE。
- 客戶可指定教練。
- 分教練的時段封鎖。
- Email/簡訊自動提醒通知（SimplyBook 有這功能，這次先不做，之後有需要再排）。
- 新生隱藏入口的正式反濫用機制（App Check/驗證碼等）——這次只靠「網址不公開」，是刻意的權宜之計，不是長期方案。
