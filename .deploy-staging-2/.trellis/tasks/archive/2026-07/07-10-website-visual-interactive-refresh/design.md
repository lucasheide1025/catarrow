# Design：官網視覺互動改版

## 設計原則

1. **每個效果都要能回答「這跟射箭/貓小隊有什麼關係」**——箭矢軌跡、靶紙同心圓、貓掌/貓咪彩蛋、命中震動感，是這次唯一允許的視覺語彙來源。不要引入跟品牌無關的泛用特效（粒子背景、彩帶、通用 3D 卡片翻轉等）。
2. **技術力展示 = 自己刻的生成邏輯，不是套函式庫**。全部用原生 Canvas 2D / SVG / CSS transform + 少量 vanilla JS 完成，比照現有 `.rv` 觀察器與跑馬燈的寫法慣例（純手刻、無依賴）。
3. **克制感優先於數量**——寧可 5 個做得精緻的細節，不要 12 個都加但很粗糙。design.md 針對每個 section 提出一個構想，實作時可以依打磨時間砍掉優先級最低的幾個（見 implement.md 的優先順序）。
4. 所有效果預設「靜態也要好看」——把互動效果拿掉，畫面依然是完整、可讀的內容，互動只是加分，不是必要條件（漸進增強）。

## 逐 Section 構想

### `#hero`（已有 `.rings`／`.target` 裝飾層，可以在此基礎加強）
- 讓 `.rings`（同心圓裝飾）隨滑鼠位置做極輕微的視差位移（`translate` 幅度控制在個位數 px，避免喧賓奪主），模擬「站在靶紙前」的空間感。
- `.target` 裝飾層：滑鼠靠近貓咪插畫時，讓同心圓其中一環短暫亮一下品牌橘（比照命中閃光的概念，但幅度要非常輕）。

### `#why`（三張貓咪卡片 `.wcard`）
- 卡片 hover 時，`.wcard::after`（目前是頂部橫線 `scaleX`）之外，可以加一個從卡片角落畫出的「爪痕刮過」SVG 描邊動畫（一筆成形，`stroke-dasharray` 技法），呼應「貓」的主題，比純色塊 hover 更有記憶點。

### `#bows`（弓種選擇）
- 若目前是靜態圖文卡片，可以加一個「選中會有箭矢輕輕震動/微幅擺動」的互動（hover 或 tap 觸發），模擬選弓的手感。**實作前務必先讀這個 section 目前的實際 DOM 結構**，構想要套進去而不是重寫整個區塊。

### `#price`（透明價格）
- 數字用**計數動畫**進場（滾動進入視窗時從 0 跑到實際金額，`IntersectionObserver` 比照 `.rv` 的觸發方式），呼應「一箭中的」的節奏感——建議數字停在最終值時，旁邊有一個極小的箭矢圖示做一次「命中」的輕微縮放回彈，不要做誇張的彩帶/爆炸效果。

### `#training`（訓練系統，本次 R2 的重點區塊）
- 手機 mockup 從單張 `assets/015.png` 靜態圖，改成**可切換的畫面預覽**：3 個小圓點/分頁指示器，點擊或滑動切換 2-3 張畫面（排行榜／打怪戰鬥／勳章圖鑑，需要準備對應截圖素材，若沒有新素材可先用現有 `assets/015.png` + 之後補拍的方式分階段做，不要因為缺素材卡住整個效果）。
- 切換動畫用簡單的 fade/slide 即可，不需要 3D 翻轉。
- `.badges` 勳章列表：目前是靜態排列，可以加上 scroll-triggered 依序「解鎖」動畫（依序淡入＋極輕微的 scale 彈跳，模擬遊戲裡「解鎖成就」的感覺），這個做法跟遊戲化系統的實際體感最貼近，也是最適合「技術力展示」的一個點。

### `#group`（團康與遊戲模式，本次 R2 的另一個重點區塊）
- 內容面：現有 4 個模式卡片（打怪模式／組隊打怪／決鬥模式／練習模式）之外，補 1-2 張新卡片介紹 App 現有但官網未提及的系統——建議優先「地下城遠征」（`Team Hunt` 之外一個更有畫面感的合作向玩法，跟團康情境契合）；「世界王」視版面空間決定要不要加，寧可少而精，不要為了湊數硬塞。
- 互動面：模式卡片 hover 時可以有一個小小的「模式圖示」動態效果（例如打怪模式的 icon 輕微攻擊震動、決鬥模式兩個圖示交錯閃現），比照現有 `.mode` 卡片的 hover transition 基礎上加強，不用整個重做。

### `#facility`（場地與師資）
- 若有場地照片輪播/畫廊，可以加輕量的 Ken Burns 式緩慢縮放效果（純 CSS `animation`，效能成本極低），營造「靜心」氛圍而非搶眼球——這個 section 的品牌語氣本來就偏沉靜，不適合太跳的效果，維持克制。

### `#reviews`（已有 `.mq-track` 跑馬燈）
- 現有跑馬燈維持，可以加 hover 暫停（滑鼠移到某則評論上跑馬燈暫停，方便閱讀），這是實用性大於視覺花俏的小改動，但體驗上很加分。

### `#faq`（`.faq-list` 手風琴）
- 現有 `summary::after` 的 `+` 展開圖示，可以在展開瞬間加一個極短的「箭矢畫過」底線動畫取代純色變化，強化品牌一致性。這個效果要非常輕量，FAQ 是資訊型區塊，不該喧賓奪主。

### `#booking` / `#visit` / `#final`
- 這三個屬於轉換型區塊（預約/地址/最終 CTA），**刻意少做視覺花招**——轉換頁面的原則是降低干擾、讓 CTA 按鈕本身最顯眼。可以維持現有按鈕 hover transition，不建議額外加生成式效果，避免分散預約行動的注意力。

## 「輕量介紹遊戲化系統」的內容範圍界線

- 只用截圖/mockup/文案說明，**不嵌入任何會連到 `src/`／Firestore 的 iframe 或元件**。`website/` 與 App 的程式碼、建置流程、部署管線維持完全獨立，這條界線不能破。
- 文案更新要對照 `docs/second_brain/features.md`／`game-systems.md` 確認 App 目前真的有的系統再寫，不要把還在規劃中的東西寫成已上線功能。
- 若要補新截圖素材，用現有 App 畫面直接截圖即可（不需要用真實會員資料，用測試帳號或教練自己的畫面截圖）。

## 效能守則（對應 prd.md 的硬性限制）

- 每個 section 最多一個「進場動畫」觸發點（沿用現有 `.rv` 的 `IntersectionObserver`，不要每個 section 各自建一個新的 observer 實例——複用同一個 observer，用不同 class 區分行為）。
- Canvas 效果（若採用）限制在小範圍元素內（例如箭矢圖示、單一裝飾層），不要做全螢幕背景 Canvas，避免行動裝置效能與電量負擔。
- 所有 `animation`/`transition` 都要有對應的 `prefers-reduced-motion` 降級規則，比照現有 CSS 中 `.rv` 的降級寫法（`opacity:1;transform:none;transition:none`）。
